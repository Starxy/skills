# `paper-pass` 设计说明

日期：2026-07-10
状态：已完成书面规格复核，进入实施

## 1. 背景

现有 `paper-reading-zh` 面向深读、工程拆解和多篇调研等多种场景，默认输出较完整。新 skill 聚焦单篇论文的第一次完整阅读：先快速建立论文心智模型，再用简明但专业的长文解释作者动机、问题缺口、核心思路、实现机制、证据、贡献和边界。

设计吸收三类优点：

- 保留 `paper-reading-zh` 的材料边界、来源锚点、事实与推断分层等证据纪律，但不复制其模式系统，也不形成运行依赖。
- 保留 FOCUS 提示词“逐节穷举提取，再清理和重组”的两阶段方法。
- 保留 `stephenturner/skill-focus` 对完整性、具体细节和直接引文的重视，但把最终输出从逐节详尽摘要改造成适合首次阅读的认知叙事。

## 2. 目标

一次成功的 `paper-pass` 应使有大领域基础、但首次接触论文细分方向的读者能够独立回答：

1. 论文研究什么问题？
2. 作者为什么要研究它？
3. 现有工作卡在哪里？
4. 论文的核心思路是什么？
5. 真正的新贡献是什么？
6. 证据支持到什么程度，主要边界是什么？

## 3. 非目标

`paper-pass` 不承担：

- 多篇论文批量摘要或横向比较；
- 完整公式推导；
- 代码复现、工程接入或实现分析；
- 逐图逐表讲解；
- 完整同行评审或系统性批判；
- 全文翻译；
- 阅读路线、是否值得读或下一步行动建议。

用户在一次调用中同时提出这些要求时，仍只完成首次阅读，并简明说明额外任务不属于本 skill。

## 4. 调用与独立性

- skill 名称：`paper-pass`
- 界面显示名：`论文初读`
- 仅由用户显式调用，不参与模型自动匹配。
- 在 frontmatter 中设置 `disable-model-invocation: true`。
- 在 `agents/openai.yaml` 中设置 `policy.allow_implicit_invocation: false`，确保 Codex 运行时也只允许显式调用。
- 与 `paper-reading-zh` 完全独立：不调用、不路由、不要求安装，也不以对方作为必需背景。
- 默认输出中文；用户明确要求时可以切换语言。

## 5. 输入边界与全文门槛

每次只处理一篇论文。输入可以是 PDF、论文网页、标题或其他可唯一定位论文的锚点，但开始解读前必须取得可读取全文。

### 5.1 全文要求

- 主论文全文是必需条件。
- 只有标题、摘要或部分正文时，先检索论文官网、开放版本、作者主页或可信仓库以补齐全文。
- 当核心方法、实验设置或关键证据被放入 Supplementary / Appendix 时，对应补充材料也成为必需条件。
- 扫描件或提取文本缺失时，尝试替代版本或可靠 OCR。
- 任一影响理解主线的正文、公式、图表或补充材料仍不可读时，列明缺失项并停止。
- 不输出“仅基于摘要”的轻量版本，也不依靠常识补齐论文内容。

### 5.2 论文识别

- 标题或引用对应多个候选时，列出候选并等待用户确认。
- 多篇输入时，要求用户选定一篇后再开始。
- arXiv、会议版或期刊版不一致时，明确当前读取版本，并以该版本为证据基础。

## 6. 总体架构

采用“精简主流程 + 类型参考文件”：

```text
skills/paper-pass/
├── SKILL.md
├── agents/
│   └── openai.yaml
└── references/
    └── paper-types.md
```

- `SKILL.md`：调用设置、全文门槛、两阶段阅读流程、输出契约、证据规则、异常处理和自检。
- `references/paper-types.md`：实验研究、算法/系统、理论、定性研究、综述/观点论文的证据镜头。
- `agents/openai.yaml`：显示名、简短说明、默认显式调用提示和禁止隐式调用的产品策略。
- 不创建脚本、资产、README 或只服务开发过程的文件。

## 7. 两阶段阅读流程

### 7.1 第一阶段：FOCUS 式穷举证据账本

按原文顺序逐节阅读，忽略参考文献、致谢、作者信息等非主体内容。对每一个关键观点记录：

- 观点或主张；
- 具体细节，包括数字、样本、方法名称、比较和条件；
- 它在论证中的作用：动机、缺口、机制、证据、贡献或限制；
- 原文直接引文；
- 精确位置锚点：章节、页码、Figure、Table、Equation、Algorithm 或可识别段落。

相近观点可以合并，但不能因为最终篇幅有限而跳过内部提取。证据账本只作为内部中间产物，不展示给用户。

### 7.2 第二阶段：按作者思考链重组

从证据账本中重建：

```text
研究动机 → 现有缺口 → 核心思路 → 实现机制 → 支撑证据 → 贡献与边界
```

重组规则：

- 区分作者明确陈述的动机与基于全文论证结构重建的设计逻辑。
- 解释性重建使用明确措辞，例如“从论文的论证结构来看”。
- 将贡献分成核心新意、关键设计、数据/系统资源和验证性结果，不等量复述作者的 contribution 列表。
- 只保留影响理解主线的内容进入最终长文，但所有关键结论应可追溯到第一阶段证据账本。
- 最终不展示第一阶段中间结果。

## 8. 复杂度与篇幅

不按页数机械决定篇幅。根据以下理解成本选择约 1000、2000 或 3000 中文字的软档位：

- 概念跳跃数量；
- 必需前置知识；
- 方法模块或论证环节数量；
- 证据类型和实验设计复杂度。

长文主体为 1000–3000 中文字。150–250 字核心总结和论文信息块不计入主体篇幅。

## 9. 最终输出契约

输出固定回答相同认知问题，但各节篇幅按论文类型动态分配。

```markdown
## 核心总结

150–250 字单段，同时交代研究问题、重要性、核心思路、主要贡献和结论。

## 论文信息

- 标题：
- 年份 / venue：
- 链接：
- 所属方向：

## 1. 作者为什么要做这项研究

## 2. 现有工作卡在哪里

## 3. 论文的核心思路

## 4. 核心思路如何落地

## 5. 论文如何证明自己的主张

## 6. 贡献与边界
```

### 9.1 各节职责

- **核心总结**：形成可独立转发的完整心智模型，不写成关键词列表。
- **作者为什么要做**：说明背景、作者明确动机和必要的解释性重建。
- **现有工作卡在哪里**：解释已有方法能力、关键假设和真正缺口，只补充主线必需的前置知识。
- **核心思路**：先用简洁、专业、低知识密度的语言建立整体认识，再说明为何可能解决缺口。
- **如何落地**：解释必要的方法结构、步骤、变量关系或研究设计。
- **如何证明**：把关键证据与其验证的主张对应起来，说明支持到什么程度及未证明什么。
- **贡献与边界**：分层呈现核心新意、关键设计、资源产出、验证结果和最重要局限。

### 9.2 公式与图表

- 默认尽量不讲公式。
- 核心贡献绕不开公式时才引入；先解释关系和作用，再明确解释所有非常见变量与参数，不做无助于主线的完整推导。
- 只选取一到两个对核心机制或主要证据必要的图表，融入对应章节。
- 不做逐图逐表讲解。

## 10. 证据规则

### 10.1 统一四层

1. 论文明确陈述；
2. 原文证据直接支持；
3. 基于全文的解释性重建；
4. 未证明、未知或未覆盖。

### 10.2 最终呈现

- 关键结论、具体数字和实验结果使用紧凑行内锚点，例如“……（§4.2，Table 3）”。
- 最终直接引文只用于定义、核心主张或容易被转述失真的内容。
- 内部证据账本中的逐点引文不全部输出。
- 未核验的 venue、年份、代码或外部背景明确标为“未核验”，不填默认值。

### 10.3 外部资料

- 论文内容、动机和贡献判断以读取版本的原文为准。
- 外部资料只用于核验元数据，或补充理解主线必需的少量前置概念。
- 外部内容使用可点击链接，并与论文内容明确区分。
- 不广泛拼接教程、博客和相关论文来替代作者论证。

## 11. 语言风格

- 保留必要学术专业性，不刻意口语化。
- 通过减少单位段落的概念数量、降低术语密度、缩短因果链跨度和稳定定义提升可读性。
- 重要术语首次出现给出中英对照，后文保持同一译名。
- 不机械翻译，不用生活类比替代精确定义；只有对应关系明确且确有帮助时才使用类比，并说明边界。
- 不写寒暄、流程预告、模板化开场或装饰性内容。

## 12. 论文类型镜头

`SKILL.md` 判断主类型后读取 `references/paper-types.md`。混合型论文选择一个主类型和一个辅助类型，不叠加全部规则。

### 12.1 实验研究

关注研究问题、假设、实验设计、对照、样本、测量和主要结果；区分相关性、机制和因果证据。

### 12.2 算法 / 系统

关注任务瓶颈、核心机制、模块职责、训练或运行流程；用基线、消融、效率和鲁棒性判断设计是否成立，不把更高分数自动等同于创新。

### 12.3 理论论文

关注问题设定、关键假设、定理结论、证明策略和适用边界；解释相比已有理论新增的保证或认识。

### 12.4 定性研究

关注研究问题、样本选择、资料获取、编码或解释过程、三角验证和可迁移性；区分参与者材料、研究者解释与进一步推论。

### 12.5 综述 / 观点

关注中心主张、材料选择方式、概念框架、证据覆盖和预测性内容；区分领域共识、作者判断与未来预测，不硬套方法—实验—结果结构。

## 13. 异常输出

停止场景只输出完成任务所需的信息，不套用正常六节结构：

- **全文缺失**：说明已取得材料、缺失材料、尝试过的检索路径，以及为何无法开始。
- **身份不唯一**：列出最可能候选及区分信息，等待用户确认。
- **多篇输入**：列出论文标题并要求选定一篇。
- **核心内容不可读**：指出不可读的章节、图表、公式或补充材料，并说明已尝试的替代来源。

## 14. 验证方案

采用 skill TDD，先运行无 skill 基线，再实现并使用相同任务复测。

### 14.1 基线失败模式

记录模型是否出现：

- 只改写摘要；
- 缺失作者动机或把推断写成作者本意；
- 平铺作者贡献清单；
- 具体结论没有证据锚点；
- 公式过度展开；
- 全文不完整仍继续解读；
- 输出滑向复现、比较或完整审稿。

### 14.2 前向测试场景

- 算法/系统论文；
- 核心贡献依赖公式的理论论文；
- 定性研究或综述/观点论文；
- 主文依赖补充材料的论文；
- 只有摘要但能检索到公开全文；
- 全文无法取得；
- 输入多篇论文；
- 论文身份不唯一。

### 14.3 验收标准

- 六个理解问题均能从输出中回答。
- 核心总结为 150–250 字单段。
- 长文主体在 1000–3000 中文字，并符合复杂度档位。
- 作者陈述与解释性重建明确分开。
- 核心贡献、关键设计、资源和验证结果分层呈现。
- 关键数字与结论有原文锚点。
- 公式和图表遵守最小充分原则。
- 全文门槛和单篇边界在停止场景中可靠执行。
- 最终输出不包含内部证据账本，不扩展到非目标任务。

### 14.4 仓库验证

- 运行 `npm run check`。
- 使用真实 skills CLI 检查 skill 可发现。
- 确认 `disable-model-invocation: true` 与 `policy.allow_implicit_invocation: false` 同时存在，使其只由用户显式调用。
- 修正前向测试暴露的最小问题并复测。
